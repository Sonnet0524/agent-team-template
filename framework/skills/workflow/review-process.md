# 代码审查流程

> 🔍 标准的代码审查（Code Review）流程

---

## 审查原则

### 核心原则

1. **发现问题是正常的** - 审查的目的是提高质量，不是指责
2. **就事论事** - 针对代码，不针对人
3. **提供解决方案** - 提出问题的同时给出建议
4. **保持尊重** - 用建设性的语气

### 审查 vs 批评

| ❌ 批评 | ✅ 审查 |
|--------|--------|
| "这代码写得太烂" | "这里可以考虑使用更高效的数据结构" |
| "你怎么能这么写" | "建议参考 xxx 的最佳实践" |
| "明显有 bug" | "第 45 行可能在边界情况下出错，建议添加检查" |

## 审查流程

### 流程图

```
收到审查请求
     ↓
快速浏览（整体印象）
     ↓
详细审查（逐行检查）
     ↓
记录问题（分类标记）
     ↓
撰写审查意见
     ↓
确认结论
     ↓
提交审查结果
```

### 步骤 1: 快速浏览

**目标**: 获得整体印象（5-10 分钟）

**检查项**:
- [ ] PR 描述是否清晰？
- [ ] 变更范围是否合理？
- [ ] 是否有明显的架构问题？
- [ ] 测试是否充分？

**决策**:
- 如果 PR 太大 → 建议拆分
- 如果描述不清 → 要求补充
- 如果架构有问题 → 先讨论架构

### 步骤 2: 详细审查

**目标**: 逐行检查代码（15-30 分钟）

**检查维度**:

#### 维度 1: 正确性
- [ ] 逻辑是否正确？
- [ ] 边界条件是否处理？
- [ ] 错误处理是否完善？
- [ ] 并发安全（如适用）？

#### 维度 2: 可读性
- [ ] 命名是否清晰？
- [ ] 函数是否简短？
- [ ] 注释是否必要且准确？
- [ ] 代码组织是否合理？

#### 维度 3: 可维护性
- [ ] 是否遵循 DRY 原则？
- [ ] 耦合度是否合适？
- [ ] 是否易于测试？
- [ ] 是否易于扩展？

#### 维度 4: 性能
- [ ] 算法复杂度是否合理？
- [ ] 是否有明显的性能问题？
- [ ] 资源使用是否合理？

#### 维度 5: 安全性
- [ ] 是否有注入风险？
- [ ] 是否有越权风险？
- [ ] 敏感信息是否泄露？

#### 维度 6: 规范性
- [ ] 是否符合编码规范？
- [ ] 是否遵循项目约定？
- [ ] 提交信息是否规范？

### 步骤 3: 记录问题

**问题分类**:

| 级别 | 标记 | 含义 | 处理方式 |
|------|------|------|----------|
| 🔴 Blocker | `[blocker]` | 必须修复 | 不修复不能合并 |
| 🟡 Major | `[major]` | 建议修复 | 最好修复，可讨论 |
| 🟢 Minor | `[minor]` | 小问题 | 可选修复 |
| 💡 Nitpick | `[nit]` | 吹毛求疵 | 仅供参考 |

**示例**:
```markdown
🔴 [blocker] 第 45 行可能导致空指针异常
🟡 [major] 建议使用常量替代魔法数字
🟢 [minor] 变量名可以更具体一些
💡 [nit] 多余的空行
```

### 步骤 4: 撰写审查意见

**结构**:

```markdown
## 总体评价
[总体评价，如：] 整体实现良好，逻辑清晰，测试覆盖充分。

## 🔴 Blockers（必须修复）
- [ ] [问题1] - [建议]
- [ ] [问题2] - [建议]

## 🟡 Major（建议修复）
- [ ] [问题1] - [建议]
- [ ] [问题2] - [建议]

## 🟢 Minor（可选修复）
- [问题1] - [建议]
- [问题2] - [建议]

## 💡 Questions（疑问）
- [问题1]
- [问题2]

## 👍 Highlights（亮点）
- [做得好的地方1]
- [做得好的地方2]

## 结论
- [ ] 批准合并（所有 blocker 已解决）
- [ ] 需要修改（有未解决的 blocker）
```

### 步骤 5: 确认结论

**批准条件**:
- 所有 🔴 Blocker 已解决或确认可以接受
- 没有未解决的安全问题
- 没有未解决的性能问题

**要求修改条件**:
- 存在未解决的 🔴 Blocker
- 存在重大架构问题
- 测试覆盖不足

### 步骤 6: 提交审查

```bash
# 使用 GitHub CLI 提交审查
gh pr review <pr-number> --comment --body "[审查意见]"

# 批准
gh pr review <pr-number> --approve

# 要求修改
gh pr review <pr-number> --request-changes --body "[审查意见]"
```

## 审查检查清单

### 通用检查项

```markdown
## 正确性
- [ ] 代码逻辑正确
- [ ] 边界条件处理完善
- [ ] 错误处理完善
- [ ] 没有明显的 bug

## 可读性
- [ ] 命名清晰（变量、函数、类）
- [ ] 函数简短（< 50 行）
- [ ] 注释准确且必要
- [ ] 代码格式统一

## 可维护性
- [ ] 没有代码重复（DRY）
- [ ] 职责单一（SRP）
- [ ] 依赖合理
- [ ] 易于测试

## 性能
- [ ] 算法复杂度合理
- [ ] 没有明显性能问题
- [ ] 资源使用合理

## 安全性
- [ ] 没有注入风险
- [ ] 没有越权风险
- [ ] 敏感信息处理正确

## 测试
- [ ] 有单元测试
- [ ] 测试覆盖关键路径
- [ ] 有集成测试（如需要）
- [ ] 测试通过

## 文档
- [ ] 代码注释充分
- [ ] API 文档更新（如需要）
- [ ] README 更新（如需要）
```

## 审查技巧

### 1. 设置时间限制

- 单次审查不超过 30 分钟
- 如果超过 30 分钟还没完成 → PR 可能太大，建议拆分

### 2. 使用工具辅助

- 自动化检查：lint、format、type check
- 静态分析：SonarQube、CodeClimate
- 测试：自动化测试套件

### 3. 关注重点

**优先关注**:
1. 正确性（逻辑、边界、错误处理）
2. 安全性
3. 性能（明显问题）

**其次关注**:
4. 可维护性
5. 可读性
6. 规范

### 4. 区分个人偏好和客观问题

**客观问题**:
- 可能导致 bug 的逻辑
- 性能问题
- 安全风险
- 违反项目规范

**个人偏好**:
- 代码风格（除非违反规范）
- 实现方式（多种方式都可行时）

## 处理审查反馈

### 作为作者

**DO**:
- ✅ 认真对待每条意见
- ✅ 不理解时主动询问
- ✅ 解释设计决策
- ✅ 及时修复问题
- ✅ 感谢审查者的时间

**DON'T**:
- ❌ 防御性回应
- ❌ 无视意见
- ❌ 延迟修复

### 作为审查者

**DO**:
- ✅ 解释为什么这是个问题
- ✅ 提供具体建议
- ✅ 承认可能出错
- ✅ 认可做得好的地方

**DON'T**:
- ❌ 只指出问题不给建议
- ❌ 坚持个人偏好
- ❌ 使用攻击性语言

## 快速审查模板

### 简单 PR（< 100 行变更）

```markdown
## 总体评价
[一句话评价]

## 问题
- [ ] [问题1] - [建议]

## 结论
- [ ] 批准 / 需要修改
```

### 复杂 PR（> 500 行变更）

```markdown
## 总体评价
[详细评价]

## 🔴 Blockers
[必须修复的问题]

## 🟡 Major
[建议修复的问题]

## 🟢 Minor
[小问题]

## 👍 Highlights
[亮点]

## 结论
- [ ] 批准 / 需要修改
```

---

## 最佳实践总结

1. **小 PR 更容易审查** - 建议 < 400 行变更
2. **自审查** - 提交前自己先审查一遍
3. **及时审查** - 收到审查请求后 24 小时内响应
4. **面对面讨论** - 复杂问题可以面对面讨论
5. **记录决策** - 重要决策记录到文档
