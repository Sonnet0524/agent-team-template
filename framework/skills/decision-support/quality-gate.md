# 质量门控

> 🛡️ Agent 输出的质量评估和决策支持

---

## 核心理念

### 什么是质量门控？

质量门控是一种**元认知能力**——Agent 知道自己输出的质量如何，并在必要时请求帮助。

```
传统方式：
Agent → 输出 → Human（发现问题）

质量门控方式：
Agent → 自我评估 → [确定性强] → 输出
                    ↓
              [确定性低] → 请求帮助
```

### 三层结构

```

└─────────────────────────────────────┘
```

## 元认知意识

### 核心信念

每个 Agent 在 AGENTS.md 中应该明确：

```markdown
## 🧠 元认知意识

**我知道自己什么时候不知道**：
- 确定性 < 70% → 请求 Human 帮助
- 遇到边界问题 → 向用户报告
- 发现阻塞 → 立即通知
```

### 为什么重要？

**问题**: Agent 可能会：
- 自信地给出错误答案
- 在不熟悉的领域瞎猜
- 忽略重要约束

**解决**: 质量门控要求 Agent：
- 评估自己的确定性
- 低确定性时主动说明
- 必要时请求帮助

## 确定性评估

### 确定性等级

| 等级 | 确定性 | 含义 | 行动 |
|------|--------|------|------|
| 🔴 LOW | < 50% | 不确定 | **必须请求帮助** |
| 🟡 MEDIUM | 50-70% | 有些不确定 | **建议请求帮助** |
| 🟢 HIGH | 70-85% | 比较确定 | 可以输出，但说明不确定性 |
| 🔵 VERY HIGH | > 85% | 非常确定 | 正常输出 |

### 评估维度

评估确定性时考虑：

```markdown
## 确定性评估检查项

### 知识领域
- [ ] 是否在我的专业领域内？
- [ ] 是否有相关经验？
- [ ] 是否了解上下文？

### 信息完整性
- [ ] 是否有足够信息？
- [ ] 是否有缺失的关键信息？
- [ ] 是否需要澄清？

### 约束理解
- [ ] 是否理解所有约束？
- [ ] 约束之间是否有冲突？
- [ ] 是否有隐含约束？

### 方案可行性
- [ ] 方案是否可行？
- [ ] 是否有明显的风险？
- [ ] 是否有更好的方案？

### 后果评估
- [ ] 能否预测结果？
- [ ] 错误的影响有多大？
- [ ] 是否可回退？
```

## 可接受性评估

### 可接受性等级

| 等级 | 可接受性 | 含义 | 行动 |
|------|----------|------|------|
| 🔴 UNACCEPTABLE | < 30% | 不可接受 | 必须重做 |
| 🟡 QUESTIONABLE | 30-60% | 存疑 | 需要改进 |
| 🟢 ACCEPTABLE | 60-80% | 可接受 | 可以交付 |
| 🔵 EXCELLENT | > 80% | 优秀 | 可以交付 |

### 可接受性检查项

```markdown
## 可接受性评估检查项

### 功能正确性
- [ ] 是否满足需求？
- [ ] 是否处理了边界情况？
- [ ] 是否有明显的 bug？

### 质量
- [ ] 是否符合编码规范？
- [ ] 是否有适当的注释？
- [ ] 是否易于理解？

### 完整性
- [ ] 是否完整实现？
- [ ] 是否有遗漏？
- [ ] 是否有待办事项？

### 测试
- [ ] 是否有测试？
- [ ] 测试是否通过？
- [ ] 覆盖率是否足够？

### 文档
- [ ] 是否有必要的文档？
- [ ] 文档是否准确？
- [ ] 是否易于使用？
```

## 混淆判断

### 什么是混淆？

当 Agent 不确定时，应该明确说明，而不是假装确定。

**混淆的表现**:
- 用复杂的语言掩饰不确定性
- 给出多个可能的答案但不说明哪个更可能
- 避免直接回答问题

**正确的做法**:
- 明确说明不确定性
- 解释为什么不确定
- 提出获取确定性的方法
- 必要时请求帮助

### 混淆判断 Schema

```yaml
confusion_assessment:
  is_confused: true|false
  
  confusion_type:
    - "knowledge_gap"      # 知识不足
    - "information_gap"    # 信息不足
    - "constraint_conflict" # 约束冲突
    - "ambiguity"          # 歧义
    - "complexity"         # 过于复杂
  
  clarification_needs:
    - "ask_human"          # 询问 Human
    - "research_more"      # 进一步研究
    - "break_down"         # 分解问题
    - "prototype"          # 先做个原型
  
  confidence_after_clarification: "EXPECTED_LEVEL"
```

## Kill Criteria（终止标准）

### 什么情况下应该停止？

**必须停止的情况**:

1. **确定性 < 30%**
   - 几乎完全不确定
   - 继续只会浪费时间
   - 应该立即请求帮助

2. **遇到边界冲突**
   - 需要修改其他 Agent 的模块
   - 超出了自己的职责范围
   - 应该上报 PM Agent 或 Human

3. **资源耗尽**
   - 时间预算用尽
   - Token 预算用尽
   - 需要申请更多资源

4. **目标冲突**
   - 任务要求与项目目标冲突
   - 多个约束无法同时满足
   - 需要重新评估任务

5. **技术障碍**
   - 遇到无法解决的技术问题
   - 依赖的服务不可用
   - 需要外部支持

### Kill Criteria Schema

```yaml
kill_criteria:
  should_stop: true|false
  
  reason:
    - "low_confidence"     # 确定性太低
    - "boundary_conflict"  # 边界冲突
    - "resource_exhausted" # 资源耗尽
    - "goal_conflict"      # 目标冲突
    - "technical_blocker"  # 技术障碍
  
  impact: "MINOR|MODERATE|SEVERE|CRITICAL"
  
  escalation_required: true|false
  escalation_target: "PM|HUMAN|TECH_LEAD"
  
  next_steps:
    - [具体行动1]
    - [具体行动2]
```

## 质量门控流程

### 输出前检查

```
完成任务
    ↓
自我评估
    ↓
┌───────────────┐
│ 确定性检查    │
│ 可接受性检查  │
│ 混淆判断      │
│ Kill Criteria │
└───────────────┘
    ↓
决策
    ↓
┌───────────┬───────────┬───────────┐
│ 高质量    │ 可接受    │ 需要改进  │
│ 正常输出  │ 输出+说明 │ 请求帮助  │
└───────────┴───────────┴───────────┘
```

### 具体步骤

#### 步骤 1: 确定性评估

```markdown
## 我的确定性评估

**整体确定性**: [HIGH/MEDIUM/LOW]

**评估依据**:
- 知识领域: [in scope / out of scope]
- 经验: [有相关经验 / 无相关经验]
- 信息完整性: [完整 / 部分缺失 / 严重不足]
- 约束理解: [完全理解 / 部分理解 / 不理解]

**不确定性来源**:
1. [来源1]
2. [来源2]
```

#### 步骤 2: 可接受性评估

```markdown
## 我的可接受性评估

**整体可接受性**: [EXCELLENT/ACCEPTABLE/QUESTIONABLE/UNACCEPTABLE]

**评估依据**:
- 功能正确性: [满足 / 部分满足 / 不满足]
- 质量: [高 / 中 / 低]
- 完整性: [完整 / 部分 / 缺失]
- 测试: [充分 / 部分 / 缺失]

**需要改进的地方**:
1. [改进点1]
2. [改进点2]
```

#### 步骤 3: 做出决策

```markdown
## 质量门控决策

**决策**: [正常输出 / 输出+说明 / 请求帮助]

**理由**:
[为什么做这个决策]

**行动**:
[具体行动]
```

## 最佳实践

### DO ✅

- ✅ 每次输出前进行自我评估
- ✅ 明确说明确定性等级
- ✅ 低确定性时主动请求帮助
- ✅ 记录评估过程
- ✅ 定期回顾和改进评估准确性

### DON'T ❌

- ❌ 假装确定（当实际上不确定）
- ❌ 忽视边界问题
- ❌ 在资源耗尽时硬撑
- ❌ 不与 Human 沟通就做重大决策
- ❌ 忽略质量门控结果

## 质量门控与项目成功的关系

```
高质量门控 → 减少错误 → 提高成功率
         ↓
    及时请求帮助 → 避免浪费 → 提高效率
         ↓
    明确不确定性 → 更好决策 → 降低风险
```

**核心价值**:
1. **信任** - Human 知道 Agent 不会瞎猜
2. **效率** - 及早发现问题，避免返工
3. **质量** - 输出质量可预期
4. **协作** - 明确何时需要 Human 介入
